# 접수관리 → 비즈핏 직접 접수 연동 계획

## 1. 현재 구조
- `접수관리.html` (브라우저, Firebase 데이터) → "단독 분류/전체 시트 분류" 클릭 시 `xlsx` 파일을 **로컬에 다운로드**.
- 다운로드한 엑셀 내용을 광고주 구글시트 `원천데이터` 탭에 사람이 직접 붙여넣음.
- 광고주 시트에 바인딩된 `Main.js`(GAS) 메뉴 "API 전송" 실행 → `ApiClient.js`의 `callWebApp_()`가 **비공개 webapp**(`WEBAPP_URL` + `WEBAPP_API_KEY`)을 호출 → webapp이 등급 검증/단가 조회/등록을 처리해 비즈핏에 접수.
- 즉 사람이 끼는 단계: **엑셀 다운로드 → 시트 붙여넣기 → 매크로 실행 → 결제 확인 팝업 승인**.

## 2. 목표
접수관리에서 분류(상품/그룹 배분)한 결과를 **엑셀 다운로드 + 시트 붙여넣기 없이** 바로 비즈핏 등록까지 이어지게 한다.

## 3. 막히는 지점 (왜 바로 fetch()로 못 붙이는지)
1. **API 키 노출** — `WEBAPP_API_KEY`는 지금 GAS(서버)에서만 쓰여서 안전함. 접수관리.html(브라우저, 게스트 계정도 들어오는 페이지)에서 직접 호출하면 클라이언트 JS에 키가 그대로 노출됨.
2. **CORS** — Apps Script 웹앱은 `application/json` POST의 브라우저 preflight(OPTIONS)를 제대로 처리하지 않아 브라우저 fetch가 막힐 가능성이 높음(지금은 서버↔서버 호출이라 문제 없었음).
3. **페이로드 매핑** — 접수관리의 상품 분배 로직(`prods`, `allocate()`, `kwForProd()`, `FIXED_PROD_IDS` 등)이 만들어내는 행 데이터를 webapp이 기대하는 `rowData` 포맷(`상품등급명/업체명/URL/MID/순위키워드/일목표/일수/예약일/검색키워드`)과 1:1로 맞춰야 함. 지금은 엑셀 시트 생성 코드(`buildProdSheet`, `buildJusuSheet`)에 이 매핑이 들어있음 — 이걸 API 호출용 JSON 빌더로 재사용 가능하게 뽑아내야 함.
4. **결제 확인 UX** — `Main.js`의 `runApi_()`는 등급별 합계/단가를 보여주고 `ui.alert(YES_NO)`로 사용자 확인을 받음. 접수관리 쪽에도 동일한 확인 모달이 필요함.
5. **로그인/세션** — `Main.js`는 광고주 도메인/아이디/비밀번호를 설정 시트에서 읽어 매번 `testWebAppConnection_`으로 로그인 확인. 접수관리는 어떤 계정으로 로그인할지(영업점 계정별로 다른 도메인/ID/PW 매핑이 필요한지) 확인 필요.

## 4. 제안 아키텍처
```
접수관리.html (브라우저)
   │ JSON POST (캠페인 분류 결과)
   ▼
higherad-auto (Cloud Run, 기존 서비스 — 인증/알림 담당)
   │ 신규 프록시 엔드포인트 추가
   │   - WEBAPP_API_KEY는 Cloud Run 환경변수로만 보관 (클라이언트에 노출 안 됨)
   │   - CORS 헤더 직접 컨트롤 가능
   ▼
비공개 webapp (WEBAPP_URL, ApiClient.js와 동일한 액션: getGradeInfo / registerGrade)
   ▼
bizfit.co.kr 등록
```
- API 키와 webapp 주소는 항상 `higherad-auto` 서버 쪽에만 있음.
- 접수관리 → `higherad-auto`/접수관리 → bizfit 사이는 **HTTPS + 내부 인증 토큰**(이미 있는 로그인 세션 재사용 또는 별도 서비스키)로만 통신.

## 5. 단계별 구현 순서 (아직 구현 안 함 — 순서 정의만)
1. **테스트 환경 분리** — 운영 `접수관리.html`을 건드리지 않고 사본(`접수관리-test.html`)에서 실험. (이번 작업에서 완료)
2. **페이로드 빌더 분리** — `buildProdSheet`/`buildJusuSheet`의 행 변환 로직에서 "엑셀 시트에 쓰는 부분"만 떼고, 같은 데이터를 `{gradeName, classType, gradeType, rowData}` JSON으로 만드는 순수 함수를 추출.
3. **`higherad-auto`에 프록시 엔드포인트 추가**
   - `POST /bizfit/grade-info` → `fetchGradeInfo_`와 동일 동작
   - `POST /bizfit/register` → `registerGrade_`와 동일 동작
   - 내부에서 `callWebApp_`와 동일하게 `WEBAPP_API_KEY`를 첨부해 webapp 호출 (키는 Cloud Run 환경변수)
   - CORS: 접수관리가 호스팅되는 origin만 허용
4. **접수관리-test.html에 "직접 접수" 버튼 추가**
   - 분류 결과 → 2단계 빌더로 JSON 생성 → `/bizfit/grade-info` 호출 → 등급/단가 응답 받음
   - 결제 확인 모달(현재 `Main.js`의 `runApi_` 확인창과 동일 정보: 건수/목표/단가/금액) 표시
   - 확인 시 `/bizfit/register` 호출 → 결과(성공/실패) 모달 표시 (기존 `buildResultHtml_`와 동일한 형태 재사용 가능)
5. **상태 반영** — 접수 성공 시 Firebase 슬롯 상태 업데이트(`등록상태`/`등록신청일` 대응 필드를 슬롯 데이터에 추가할지 결정).
6. **테스트**
   - 비즈핏 테스트 계정으로 1~2건 소량 등록 시도 → 응답/에러 케이스 확인 (등급 오류, 과거 예약일 스킵, 결제 실패 등 `Main.js`에 있던 모든 분기 동일하게 재현되는지 확인)
   - 게스트 계정으로는 이 기능이 안 보이는지 확인 (현재 게스트는 순위/유형 일부가 가려져 있음)
7. **점진적 전환**
   - 처음엔 "엑셀 다운로드" 버튼은 유지하고 "직접 접수" 버튼을 나란히 추가 (병행 운영)
   - 안정화되면 엑셀 다운로드는 백업/감사용으로만 남기고 기본 흐름을 직접 접수로 전환
8. **운영 반영** — `접수관리-test.html` 검증 끝나면 `접수관리.html`에 반영하고 임시 사이드바 메뉴/사본 파일 제거.

## 6. 미해결 질문 (구현 전 확인 필요)
- 영업점(대행사)마다 비즈핏 도메인/로그인 계정이 다른가? 그렇다면 접수관리에서 어떻게 계정을 선택/매핑할지.
- 결제(과금) 책임은 누가 누르는 시점에 발생하는가 — 접수관리 담당자가 눌러도 되는 액션인지 운영 정책 확인.
- `higherad-auto`에 외부(bizfit) 호출 권한/네트워크 정책 추가가 필요한지.
